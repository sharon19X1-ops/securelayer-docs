Q2 — "How does it manage AI agents beyond monitoring filesystem and network activity?"

Answer: Semantic Enforcement Layer
Core differentiator
This is the most important technical question. It goes to the heart of what is different about AI agent security vs. traditional EDR.

Traditional EDR (CrowdStrike, SentinelOne): monitors at the OS layer — what process is running, what file it touched, what network call it made. It knows an AI agent is running because it sees claude-code or cursor as a process. It cannot tell you what that agent was instructed to do or what skill it invoked. A shell-exec-sudo call looks the same as a legitimate terminal command.

SecureLayer operates one level up — at the semantic layer. The Hook agent sits between the AI agent and the action it's about to take. It intercepts at the IDE plugin layer (VS Code extension, JetBrains plugin, Cursor integration) and at the MCP server interface — before any filesystem write, process spawn, or network call is made. It sees the intent (the skill being invoked, the tool call being made, the permission being requested) and applies a policy decision before the action executes.

What this means in practice:

→ Skill-level enforcement: when env-reader is invoked for the first time, SecureLayer scores it across 10 dimensions (credential access: 10/10, permission scope: 9/10, reversibility: 9/10) and auto-blocks before the agent reads a single environment variable.
→ Office hours enforcement: a policy rule says "no AI agent actions outside 08:00–18:00." The Hook intercepts any agent invocation at 11pm and returns a policy-denial to the IDE — the developer sees "blocked by policy" before any code executes.
→ MCP server interception: when a Claude Code agent calls the browser-fetch MCP tool (network egress: 9/10, supply chain risk: 8/10), SecureLayer intercepts the tool call, scores it, and either approves, holds for MSSP review, or blocks — before the HTTP request leaves the machine.

✓ Key phrase to use with the MSSP: "We don't monitor what happened — we enforce what's allowed to happen. The difference is the enforcement layer sits between the AI agent's intent and the system action, not after the action."
⚠ Internal: Phase 1 honesty. The runtime Hook (IDE plugin + MCP interceptor) is a Sprint 3 deliverable (Weeks 7–9). Phase 1 Sprints 1–2 operate at the manifest/declaration level — we detect what agents and MCP servers declare they can do, score it, and surface it. Runtime interception (blocking mid-action) comes in Sprint 3. Be clear with the MSSP: Phase 1 pilot starts with discovery + policy configuration, enforcement follow on in Sprint 3.

Q3 — "I am baffled by the skills detection capability"

Answer: Explain the Model
Needs clearest explanation
The MSSP is not confused — they're asking for the technical definition. "Skills" is SecureLayer's term for the individual capabilities an AI agent can invoke. This maps directly to what MSSPs already know as "tool calls" or "agent actions."

A "skill" in SecureLayer = a discrete capability an AI agent can invoke. Examples: browser-fetch (fetch a URL), git-push (push code to a repo), env-reader (read environment variables), shell-exec-sudo (run a shell command with elevated privilege). These are the atomic actions that AI agents like Claude Code, Cursor, and GitHub Copilot are instructed to perform.

The SecureLayer skill registry maintains a real-time inventory of every skill observed across all tenants — when it was first seen, last used, how many times invoked, which agent type invoked it, and its current policy status (auto-approved / pending review / blocked).

Each skill is scored across 10 risk dimensions:

The MSSP approves or blocks from the dashboard or directly from the ticket — one click. Approved skills are remembered per-tenant; they never resurface unless the skill's behaviour changes.

Q4 — "Can it leverage RMM to run software, block usage, run scripts with privilege over thousands of endpoints? Self-protect against compromised endpoints and false positives?"

Answer: Yes to RMM · Nuanced on Self-Protection
RMM answer: Yes
The MSSP guessed correctly on RMM leverage. This is a core design pillar of the v6 architecture — not an afterthought. The platform is built for MSSP-scale deployment via the tools MSSPs already operate.

RMM deployment is native in the architecture (v6). The Hook agent deploys via NinjaOne, Datto RMM, N-able N-central, Kaseya VSA, or ConnectWise RMM. The deployment script runs with system privilege, installs the Hook as a system service (launchd / systemd / Windows Service), deploys the IDE extension (VS Code via code --install-extension, JetBrains plugin JAR, Cursor integration), and reports back to the RMM within 60 seconds of success. One RMM script, thousands of endpoints — exactly what the MSSP asked about.

RMM-visible health reporting: every endpoint shows heartbeat status (alive / not responding), hook version, active/paused/error state, and last-event timestamp in the RMM console. If a hook goes offline, the MSSP sees an alert in NinjaOne or ConnectWise — the same alert channel they already monitor.

On self-protection against compromised endpoints: this requires an honest, layered answer.

→ Phase 1 (no endpoint hook): no endpoint-level self-protection concern exists because the manifest scanner runs cloud-side. The MSSP console and all policy enforcement happen server-side — nothing on the endpoint can disable what it cannot reach.
→ Phase 2 (runtime Hook): the Hook binary is signed (code-signing certificate), the system service runs under a dedicated service account (not a developer account), and macOS TamperProtection / Windows Protected Process can be leveraged. If the Hook is killed by a compromised process, the RMM health report immediately shows "not responding" — triggering an MSSP alert. The cloud-side policy is the authoritative source; the hook is a local enforcer. A compromised endpoint that silences the hook cannot forge an "all-clear" to the server.

✓ Key phrase for the MSSP: "The policy is enforced cloud-side, not just on the endpoint. If the local hook is silenced, the MSSP dashboard shows the gap — it doesn't silently fail safe. An offline hook is a visible alert, not an invisible bypass."

On false positives and operational disruption — this is the most operationally sensitive concern:

→ The 10-dimension scoring model with tunable auto-approve threshold is specifically designed to prevent analyst fatigue. Low-risk skills (score ≤ 3.0) never generate a ticket.
→ The MSSP can tune the threshold per tenant — a more risk-tolerant tenant has a higher auto-approve ceiling; a regulated financial client has a lower one.
→ Break-glass for developers: a blocked developer can raise a ticket directly from their IDE ("request approval") which routes to the MSSP queue with full context. The MSSP approves in one click; the approval persists for that skill + that tenant forever.
→ Office Hours enforcement is off by default. The MSSP enables it per tenant, per policy — it is never automatically applied across a fleet.

⚠ Internal concern: The MSSP's false-positive worry is legitimate and should be treated as a first-class engineering requirement, not a sales objection to manage. If Phase 1 generates noisy tickets on legitimate developer workflows, we lose the design partner. The auto-approve threshold must be conservative (default: block only score ≥ 8.0) and the MSSP must be able to tune it from day one. This is a Sprint 1 product decision — not a Sprint 3 refinement.

C2

False positive risk is the #1 operational threat to the design partnership

Sprint 1 · Product Decision
The risk
If the skill scoring model auto-blocks a legitimate developer workflow (e.g. a git-push flagged because it runs a pre-commit shell hook), the MSSP gets an angry client call within hours. One false positive at pilot stage destroys the design partner relationship.
Required before Sprint 1 ships
(a) Default auto-block threshold must be conservative: ≥ 8.0 only. (b) Auto-approve threshold must be clearly defined: ≤ 3.0. (c) The pending review band (3.1–7.9) must generate a ticket but NOT block execution until the MSSP explicitly approves blocking. (d) Developer break-glass must exist from day one.
Owner: Engineering Lead — threshold values must be locked in the Sprint 1 spec, not left as config defaults to be tuned later.
C3

MCP server interception (runtime) requires a proxy architecture — not trivial

Phase 2 · Architecture Decision
The technical challenge
MCP servers communicate via stdio (local) or HTTP. Runtime interception without a hook agent requires either (a) a local proxy that sits between the IDE and MCP server, or (b) an IDE extension that wraps MCP calls before they leave the plugin. Option (a) is transparent to the agent; option (b) requires per-IDE plugin development. Neither is trivial.
Phase 1 scope (safe)
Phase 1 only does manifest-based MCP detection — parse .mcp.json and tool declaration files, score declared capabilities. No runtime interception in Phase 1. Phase 2 decides between proxy vs. IDE-extension approach based on what the design partner's environment looks like (which IDEs, which MCP servers).
Owner: Engineering Lead — architecture decision for Phase 2 proxy vs. IDE extension must be made at the Sprint 3 design review, not before. Ask the MSSP which IDEs their developer clients use in the follow-up call.
C4

Self-protection against compromised endpoints is a Phase 2 problem only

Phase 2 · Be Honest
What we can say now
Phase 1 has no endpoint hook, so no endpoint-level self-protection risk. Cloud-side enforcement is inherently resilient — an endpoint cannot forge an approval. This is actually a strength of the Phase 1 architecture: no footprint to compromise.
What Phase 2 must resolve
Code-signed binary, system service account (not developer account), TamperProtection API usage where available, hook-offline alert via RMM within 60 seconds. The offline alert IS the self-protection mechanism — silence is visible, not invisible. This must be specified before the Phase 2 runtime sensor design starts.
Owner: Engineering Lead — Phase 2 spec must include binary signing + TamperProtection requirements. Sharon to communicate Phase 1 advantage (no footprint = no endpoint risk) in the MSSP reply.
C5

We don't know their RMM platform or PSA — critical unknowns before pilot scoping

Discovery · Ask Now
Why this matters
The architecture supports NinjaOne, Datto, N-able, Kaseya, ConnectWise RMM — but each has a different deployment script. PSA adapters exist for ConnectWise, Autotask, HaloPSA, Syncro — but building the wrong one first delays the pilot by weeks. We have one guess: the MSSP mentioned "RMM agents" without specifying which.
What to ask
"Before we scope the pilot, three quick questions: (1) Which RMM platform are you running? (2) Which PSA? (3) How many developer endpoints do your target SMB clients have on average?" These three answers define which integration to build first and exactly how fast the pilot can deploy.
Owner: Sharon — include these three questions in the reply email. Do not scope the pilot without them. The wrong RMM assumption wastes a sprint.
