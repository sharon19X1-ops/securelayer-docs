The Concept — One Page

Foundation
The Problem

AI coding agents — Claude Code, GitHub Copilot, Cursor, MCP servers — are deployed across SMB environments. They have broad file-system, shell, and API access. They run during and after business hours. No MSSP has visibility into them. No tool governs what they can do. Auditors are already asking for evidence of AI agent access controls. Active attacks via stolen credentials and prompt injection are documented in enterprise incident reports.

The Solution

SecureLayer is the enforcement layer between AI agents and the environments they reach. Phase 1 delivers: real-time risk scoring per agent skill (10-dimension model), MCP server detection (manifest scan), office hours enforcement (block outside defined windows), and a compliance artifact (audit-ready access log). Phase 2 adds a lightweight runtime sensor for mid-session threat detection.

Positioning

EDR for the Agentic Era. The first security product that enforces policy on what AI agents can do — not just logs what they did.

Primary Channel

MSSP vCISO. Per-tenant add-on, white-label ready, invoiced to SMB clients at QBR. $20–30/tenant/month flat — no negotiation until 10 customers.

PLG Supporting Wedge

GitHub OAuth scan → free AI agent + MCP inventory → enforce? upgrade trigger. Feeds MSSP inbound. Not a competing channel.

Phase 1 MVP — Weeks 1–12

Build Now · No Endpoint Agent

Phase 1 is entirely API-based — no endpoint agent required. It discovers what is running, scores risk, surfaces compliance gaps, and enforces policy via the MSSP's existing RMM/PSA stack. The API is designed from day one with a clean hook for the Phase 2 runtime sensor so no retrofit is needed when MSSPs want to upgrade.

Phase 2.1
Sensor deploy

Lightweight runtime sensor — RMM-deployed, zero friction upgrade

Existing MSSP flips the "Deep Inspection Mode" toggle in the Phase 1 dashboard. Conductor pushes the sensor to all managed endpoints via the MSSP's existing RMM. The Phase 2 API hook (stubbed in Sprint 3) activates. No new rollout conversation — the MSSP has already agreed to the scope during Phase 1 onboarding. Sensor is lightweight by design: reads agent session metadata, intercepts MCP traffic, reports to SecureLayer cloud. No kernel-level access required.

RMM toggle upgrade Zero re-architecture Phase 1 API hook activated Lightweight — no kernel access
Phase 2.2
MCP behavioral

Full MCP behavioral analysis — traffic inspection + anomaly detection

Phase 1 detected MCP servers from manifests. Phase 2 inspects what those servers actually do: traffic patterns, tool call frequency, data volume, unusual access sequences. Anomaly scores are added as a new dimension to the existing 10-dimension skill score model. This is the most defensible position — MCP traffic is invisible to all existing security tools (CrowdStrike, Palo Alto, CASB). SecureLayer is the only layer that sees it.

MCP traffic inspection Anomaly detection 11th risk dimension Invisible to all incumbents

These are the documented active threats from the market evidence section. Phase 2 is the only product that catches them.

Decision Log — D1 Through D5

D1

GTM scope: MSSP primary with PLG as supporting wedge — not parallel motions

Selective Expansion · Locked
What was decided
SELECTIVE EXPANSION. MSSP channel is the primary revenue motion. PLG (GitHub OAuth scan) exists to generate inbound signal and real usage data — it is not a competing revenue stream. All engineering and marketing energy points at MSSP first.
What this prevents
Building two parallel go-to-market motions with equal resource allocation before either is validated. PLG is a wedge and proof-of-product, not a business model of its own until MSSP motion is proven.
Trigger to revisit: PLG self-serve reaches 50 paying companies before 10 MSSP partners close. At that point, reassess whether PLG deserves its own motion and resource allocation.
D2

Pricing: $20–30/tenant/month flat — hold for market penetration

Market Penetration · Locked
What was decided
Hold $20–30/tenant/month flat. No tiering, no negotiation with first 10 MSSP customers. The price is a market penetration play — getting in the door and establishing reference customers matters more than extracting maximum margin right now.
What this prevents
Two-tier pricing (visibility-only vs. enforcement+compliance) adds friction to the pitch before you have proof the basic model works. MSSP has to choose. Early discovery calls confirmed compliance is equally valued — bundle it at flat price.
Trigger to revisit: After 10 paying MSSP customers, review whether a Phase 2 premium tier ($45–60) for runtime sensor + behavioral analysis is warranted. Discovery calls from those 10 will define the ceiling.
D3

MCP detection: manifest scan in Phase 1, full behavioral analysis in Phase 2

Phased · Locked
What was decided
MCP server detection is included in Sprint 1 scope — but manifest-based only. Parse .mcp.json, MCP config files, and tool declaration manifests to identify what servers are present and what tools they expose. Score them on the existing 10-dimension model. Full traffic inspection and behavioral anomaly detection are Phase 2.
Why this matters
MCP servers are the most defensible threat surface — they are invisible to all existing security tools. Even manifest-level detection in Phase 1 gives SecureLayer a unique claim no incumbent can match. Full behavioral analysis (Phase 2) is the moat.
Engineering note: Do not delay Sprint 1 trying to build Phase 2 MCP inspection. Manifest scan is the wedge. Full behavioral analysis requires the Phase 2 runtime sensor. Build in that order.
D4

Runtime sensor: Phase 2 component — Phase 1 API must include clean hook stub

Architecture Lock · Sprint 3
What was decided
Phase 1 has no endpoint agent. Phase 2 introduces a lightweight runtime sensor. The Phase 1 API must be designed with a clean, documented hook for the Phase 2 sensor from day one (wired in Sprint 3). When an MSSP wants Phase 2, they flip a toggle — no re-architecture, no new sales conversation. The sensor deploys via their existing RMM.
What Phase 2 sensor enables
Mid-session attack detection: credential theft patterns, prompt injection sequences, privilege escalation. These are the documented active threats. No Phase 1 product can catch them — requires runtime presence. The sensor is lightweight by design (no kernel access, session metadata + MCP traffic only).
Sprint 3 requirement: The Phase 2 API hook stub must be implemented in Sprint 3 alongside Office Hours enforcement. The "Deep Inspection Mode" toggle must appear in the Phase 1 UI, greyed out, labelled "Phase 2." This is not optional — it signals the roadmap and reduces Phase 2 upgrade friction with every MSSP from day one.
D5

Platform partnerships: defer until 10 MSSP customers — use traction as proof point

Phase 2 Only · Locked
What was decided
Anthropic, Microsoft (GitHub/Azure), and GitHub should eventually be distribution channels — recommending SecureLayer to their enterprise and MSSP customers. This is a Phase 2 motion only. Do not pursue vendor partnerships in Phase 1. They distract from the MSSP motion and there are no proof points to bring to the table.
The Phase 2 pitch
"We protect your enterprise customers' deployments of your agents. We currently secure [X] MSSP environments covering [Y] SMBs running Claude Code / Copilot / Cursor. A recommended integration from your platform means your customers can adopt agents without security risk." That conversation only works with 10+ live customers.
Trigger to act: 10 paying MSSP customers live on Phase 1. Initiate partnership conversations with Anthropic and Microsoft simultaneously. Use customer logos and incident-prevention data as the proof point. The pitch is distribution, not technical integration.

Competitive Position — The Enforcement Gap

Favorable Window · Time-Bounded

Every major security vendor is acquiring AI agent visibility capabilities. None has shipped enforcement. That gap is SecureLayer's entire reason to exist — and it is time-bounded to approximately late 2026 / mid-2027 based on M&A integration timelines.

CompetitorWhat They HaveWhat They LackSecureLayer Advantage
CrowdStrike / SentinelOne (XDR) Process telemetry, AI executable monitoring Semantic enforcement — cannot block based on what an agent is doing, only that it's running Skill-level risk scoring + session-level block policy + MCP behavioral analysis
Palo Alto / Prisma (CNAPP) Cloud AI workload inventory, model catalogues Developer endpoint enforcement; no MSSP multi-tenant model MSSP-native multi-tenant, RMM/PSA integration, Phase 1 requires no endpoint agent
Microsoft Purview / Defender DLP for prompts, M365 Copilot policy Non-Microsoft agents (Claude Code, Cursor, MCP servers) — completely blind to them Cross-agent coverage regardless of vendor. Claude, Cursor, Copilot — all in one pane.
Observability startups (being acquired) Agent inventory, usage logging, dashboards Enforcement — they alert but cannot block; no compliance artifact; no MCP behavioral analysis Block + alert + audit log + MCP inspection in one enforcement layer
CASB vendors (Netskope, Zscaler) Network-level AI SaaS usage visibility Agentic behavior — autonomous multi-step actions bypass network controls entirely Enforcement at the agent policy layer, not network layer. MCP traffic is invisible to CASB.
Strategic Window

The enforcement gap stays open until approximately late 2026 to mid-2027. SecureLayer must reach 20+ paying MSSP partners and establish the enforcement standard before platform vendors bundle a competing feature. The MCP behavioral analysis layer (Phase 2) is the long-term moat — MCP traffic is invisible to every existing security tool. That defensibility compounds as MCP adoption grows.
